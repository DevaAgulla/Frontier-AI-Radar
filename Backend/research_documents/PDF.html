Section 1

The Problem — What We Were Asked to Build

The Frontier AI Radar hackathon challenge requires building a system that autonomously monitors the global AI ecosystem and delivers a daily intelligence digest. The scope covers four distinct intelligence domains, each requiring different crawling strategies, reasoning capabilities, and data sources.

Core Mission

Track competitor releases, foundation model provider updates, latest research publications, and HuggingFace benchmarking results — then produce an executive + technical PDF digest delivered by email, with a web UI for management.

The Core Justification

The problem requires adaptive intelligence — not mechanical execution. The web is unstructured, sources change format, claims need verification, and relevance requires reasoning. A pipeline processes what it expects. An agent handles what it finds.

Section 3

The 5 Agentic Properties — How Each Is Satisfied

For a system to be genuinely agentic — not just labeled agentic — it must satisfy five core properties. Here is exactly how each property is implemented in our system.

① Goal-Driven Behavior

Definition: Each agent has an explicit goal it works toward — not just a function to execute.
Implementation: Every agent's system prompt opens with a clear goal statement. The Mission Controller holds the overall mission goal in RadarState.mission_goal. The Strategy Planner converts this into daily sub-goals stored in RadarState.strategy_plan. Every intelligence agent reads its sub-goal before starting its ReAct loop.

② Autonomous Decision Making

Definition: The agent decides its next action based on what it observes — not a hardcoded sequence.
Implementation: Every agent uses LLM reasoning (Claude) to select its next tool call. When the Competitor Intelligence Agent finds a thin blog page, Claude decides to call fetch_headless(url). When Research Agent gets low-relevance results, Claude decides to call search_semantic_scholar() with a refined query. No if/else in agent code — all decisions are LLM-driven.

③ Tool Usage

Definition: Agents act on the world through a defined set of tools they can invoke.
Implementation: 21 tools defined in core/tools.py. Each tool is a callable with a clear docstring that Claude reads to understand what it does. Tools are passed to Claude via the Anthropic API's tool-use interface. Claude returns a structured tool call — the framework executes it — the result returns to Claude for the next reasoning step.

④ Memory / Context Awareness

Definition: Agents retain and use context — within a run, across runs, and about known entities.
Implementation: 3-layer memory system. Short-term: LangGraph TypedDict state shared across all agents in a single run. Long-term: PostgreSQL stores historical findings — Strategy Planner reads this to adjust today's priorities. Entity memory: ChromaDB vector embeddings of known organizations, models, and benchmarks — every agent reads entity context before summarizing a finding.

⑤ Reasoning Before Acting

Definition: The agent thinks through its situation before taking any action — not just pattern-matching to a response.
Implementation: The ReAct loop enforces this structurally. Claude is prompted: "Reason step by step: What did you find? Is it good enough? What should you do next?" before any tool is called. This is not cosmetic — Claude's reasoning directly determines whether it loops, emits a finding, or escalates to the Mission Controller.

ReAct (Reason + Act) is the cognitive loop that every agent in our system follows. It is what separates genuine agents from labeled functions. Here is how it works in detail.

StepWhat HappensExample (Competitor Agent)

② Read MemoryReads entity memory for context about this competitor and last seen stateReads: "Last Anthropic update was Claude 3.5 Sonnet on [date]"
③ ReasonClaude thinks: what do I know? What do I need? What's the best first action?"Blog is JS-rendered per entity memory. Should start with headless fetch."

⑥ Evaluate QualityClaude assesses: is this sufficient? Confident? Or needs more investigation?"Found 1 new post. Title mentions 'Claude 4'. Need changelog to confirm release details."
⑦ Write MemoryUpdates short-term state with what was found. Updates entity memory if new info discovered.Writes: page snapshot hash, current iteration count
⑧ DecideLoop (fetch more), Emit (confident finding ready), or Escalate (cannot resolve — flag to Mission Controller)Loops: calls fetch_webpage('docs.anthropic.com/changelog') for spec details
Why Max 3 Iterations?

Each agent is capped at 3 ReAct iterations per source. This prevents infinite loops while still allowing meaningful adaptive behavior. If an agent cannot reach confidence ≥ 0.6 in 3 iterations, it emits a low-confidence finding with an "unverified" flag rather than blocking the pipeline.

Section 5

The 6-Layer Architecture — All 12 Agents Explained

The system is organized into six functional layers, each with a distinct responsibility. Agents within the same layer run in parallel where possible. Layers execute in sequence with shared state passing between them.

Layer 1 — Mission Control (1 Agent)
🎯
Mission Controller
Sole authority on mission success
Oversees the entire daily run from start to finish. Monitors the progress of all 11 other agents by reading shared state. When it detects that findings in any domain are below threshold, it autonomously triggers additional investigation rounds. Has the authority to re-dispatch agents, extend iteration limits, or mark the run as partial-complete if a domain is entirely unavailable.

🗺️

Sets today's intelligence priorities
Reads yesterday's findings from long-term memory before each run. Reasons about what was significant, what topics are trending, and what domains need extra depth today. Writes a structured strategy plan into shared state that all Layer 4 agents read before starting. For example: if three competitors released models yesterday, it increases crawl depth for model provider sources today.

Benchmark Intelligence
HuggingFace leaderboard tracker
Monitors HuggingFace leaderboards, trending models, and evaluation datasets daily. Detects meaningful rank changes, new model families entering the top 10, and SOTA shifts. Layer 5 — Validation & Ranking (2 Agents)
✅
Verification Agent
Independent claim validation
Receives all verification tasks dispatched by Model Intelligence. For each task, independently fetches the relevant HuggingFace leaderboard and finds the actual score for the claimed model and benchmark. Returns one of four evidence-backed verdicts: Verified / Partially Verified / Contradicted / Not Found. These verdicts are included in the final PDF with supporting evidence.
ReAct LoopTool CallingMemory R/WIssues Verdicts
🏆
Ranking Agent
Dedup, cluster, score all findings
Applies the Impact Score formula across all findings from all four intelligence agents. Performs two-stage deduplication: first by content hash (exact), then by embedding cosine similarity above 0.92 (semantic). Clusters findings by topic and entity. Produces a ranked, deduplicated, clustered list ready for the Digest Agent.
ReAct LoopTool CallingMemory R/W
Layer 6 — Synthesis & Delivery (3 Agents)
📝
Digest Agent
Narrative synthesis + dual output
Synthesizes ranked findings into a coherent daily intelligence narrative. Identifies the dominant theme across all findings (e.g., "context window expansion wave", "pricing competition intensifying"). Critically — if Claude evaluates its own output and finds it reads as a plain list rather than a story, it autonomously rewrites it. Produces both JSON (for storage) and Markdown (for PDF rendering).
ReAct LoopTool CallingMemory R/WSelf-Rewrites
📄
Report Generator
Branded PDF renderer
Renders the final branded PDF digest from the Digest Agent's Markdown output. Structure: Cover Page → Executive Summary (7 items) → Competitor Intelligence → Model Provider Updates → Research Highlights → Benchmark Movements → Verification Verdicts → Appendix with source citations. Uses WeasyPrint HTML→PDF for polished output.
Tool CallingMemory R/W
📬
Notification Agent
Email, dashboard, archive
Distributes the completed digest via three channels: email (inline executive summary + PDF attachment to configured recipients), web dashboard update (latest run status and top findings), and digest archive (stored for search and historical comparison). Implements retry queue with exponential backoff for email delivery failures.

Section 7

The 3-Layer Memory System

Memory is what separates an agent from a stateless function. Our system implements three distinct memory layers, each serving a different temporal and contextual purpose, wired into every agent's ReAct loop.

⚡ Short-Term Memory
Stores all findings, tool results, and observations from the current run only. Agents read this before acting to avoid re-fetching content already retrieved by another agent in the same run. Cleared automatically after each run completes.
Implementation: LangGraph TypedDict State (in-process, zero latency)
📚 Long-Term Memory
Persists historical findings, past digests, content hashes, and trend baselines across runs. Strategy Planner reads this every morning to calibrate today's priorities. Digest Agent reads this to generate "What changed since yesterday." Enables incremental diff — only re-summarize pages that actually changed.
Implementation: PostgreSQL with pgvector index for similarity queries
🏷️ Entity Memory
Tracks known organizations, AI models, benchmarks, and papers as vector embeddings. Every agent reads relevant entity context before summarizing a finding — enabling accurate statements like "This is Anthropic's 4th major model release this year." Semantic search enables fuzzy entity matching.
Implementation: ChromaDB (vector embeddings) + Postgres (structured metadata)

Section 10

Impact Scoring Formula

Every finding produced by every intelligence agent is scored by the Ranking Agent using this formula. The score determines position in the daily digest. Weights are configurable in config/scoring.yaml.

Impact Score  =  0.35 × Relevance  +  0.25 × Novelty  +  0.20 × Credibility  +  0.20 × Actionability
FactorWeightHow It's ScoredHigh Score Criteria
Relevance35%Claude scores 0–1 based on entity + topic match against org prioritiesMatches tracked entities, covers eval / agents / safety / multimodal
Novelty25%Days since similar finding last seen (from long-term memory)Not seen in last 7 days. Genuinely new development.
Credibility20%Source classification: official primary > official secondary > repostOfficial company blog, official docs, peer-reviewed paper
Actionability20%Claude scores 0–1 based on presence of concrete changesIncludes API change, pricing update, eval result, code/tool release

Why This Wins the Hackathon

Every architectural decision maps directly to one of the three judging criteria: Innovation, Clarity of Thought, and Execution Excellence with Results.

Judging CriterionHow Our System Scores
Innovation Cross-agent SOTA verification is genuinely novel. The Mission Controller re-planning at runtime is not a standard pipeline pattern. The Digest Agent self-rewriting its own output is a demonstration of self-reflection — a cutting-edge agentic capability. The 3-layer memory system enables intelligence that improves over time, not just within a single run.
Clarity of Thought The 6-layer architecture has a clean separation of concerns — each layer has one responsibility, each agent has one goal. The Impact Score formula is visible and explainable — judges can see exactly why Item #1 ranked above Item #2. LangSmith traces show every reasoning step Claude took. The system can be explained at any level of depth: one sentence, one diagram, or 12 sections.
Execution Excellence The demo script is a 60-second end-to-end flow: Add URL → Run Now → Watch 12 agents execute in the LangSmith trace → See ranked findings → Download branded PDF → Email received. Every component is functional, not mocked. Partial failure tolerance means the demo does not crash if one source is unavailable. The web UI makes the system accessible to non-technical judges.
